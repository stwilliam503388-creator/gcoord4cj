## 算法差异分析

两个版本在核心数学公式上完全一致（均采用 GB/T 20257 国测局偏移多项式），
但在 **GCJ02 → WGS84** 反向求解策略上存在显著差异：

| 项目 | JS gcoord | cj-gcoord (仓颉版) |
|------|-----------|-------------------|
| GCJ02 → WGS84 算法 | 收敛循环（while \|dx\| > 1e-6） | 固定 **30 次**迭代 |
| 实际迭代次数（中国境内） | **平均 ~3 次**（最少 2，最多 3） | 固定 **30 次** |
| 逆向精度 | < 1×10⁻⁶ 度（≈ 0.1 m） | < 1×10⁻⁷ 度（≈ 0.01 m） |
| 算法工作量比 | 基准 1× | ≈ **10×**（针对 GCJ02→WGS84） |
| 其他转换路径 | 无差异 | 无差异 |

**结论**：仓颉版在 GCJ02 → WGS84 方向精度高出约一个数量级，代价是运算量约为 JS 版的 10 倍。
若追求吞吐量可将迭代次数从 30 次减至 5 次，仍可保持 < 1×10⁻⁶ 度精度。

## 仓颉版性能估算

> 由于当前 CI 环境未安装 cjc 编译器，下表为基于以下依据的理论估算：
> 1. 仓颉编译为 LLVM 原生二进制，数学密集型循环通常比 V8 JIT 快 **3–6×**；
> 2. GCJ02→WGS84 的 30 次固定迭代 vs JS 的 ~3 次收敛迭代，额外工作量约 **10×**；
> 3. 数组分配：仓颉每次调用创建 Array<Float64>，JS 使用裸数组字面量，开销相近。

| 转换路径 | JS 实测单次耗时 | 仓颉估算单次耗时 | 估算吞吐量 | 说明 |
|---------|--------------|----------------|----------|------|
| WGS84 → GCJ02 | 1,339 ns | ~280–450 ns | ~2,200–3,600k ops/s | 原生约快 3–5× |
| **GCJ02 → WGS84** | 1,825 ns | **~3,000–3,600 ns** | **~280–330k ops/s** | 30 迭代 vs 3 迭代，净慢约 2× |
| GCJ02 → BD09 | 1,088 ns | ~220–360 ns | ~2,800–4,500k ops/s | 原生约快 3–5× |
| BD09 → GCJ02 | 1,098 ns | ~220–360 ns | ~2,800–4,500k ops/s | 原生约快 3–5× |
| WGS84 → BD09 | 1,537 ns | ~500–750 ns | ~1,300–2,000k ops/s | 两步复合，原生仍快 2–3× |
| **BD09 → WGS84** | 1,917 ns | **~3,200–4,000 ns** | **~250–310k ops/s** | 含 30 次逆向迭代，净慢约 2× |
| WGS84 → EPSG3857 | 996 ns | ~200–330 ns | ~3,000–5,000k ops/s | log/tan，原生约快 3–5× |
| EPSG3857 → WGS84 | 979 ns | ~200–330 ns | ~3,000–5,000k ops/s | atan/exp，原生约快 3–5× |
| WGS84 → EPSG4326 (same) | 9 ns | ~1–3 ns | >300,000k ops/s | 直接返回，原生快 3–9× |

> **如需获得实测数据**，请在安装了 cjc 0.53.4 的环境中执行：
> ```bash
> cd bench && cjpm build && cjpm run
> ```

## 优化建议

### 针对仓颉版

1. **减少 GCJ02→WGS84 迭代次数**：将固定 30 次改为 5 次，精度仍 < 5×10⁻⁷ 度，
   吞吐量可提升约 6×，超过 JS 版本。

2. **复用输出数组**：`transform()` 每次调用均分配新 `Array<Float64>`。
   若提供 in-place 变体 `transformInPlace(coord, from, to)`，可减少 GC 压力。

3. **批量 API**：新增 `transformBatch(coords, from, to)` 接受坐标数组切片，
   利用 SIMD/向量化（Cangjie 内联汇编或 intrinsics）进一步加速。

### 针对 JS 版

1. **TypedArray 版本**：用 `Float64Array` 替代普通数组，减少装箱开销，
   GCJ02→BD09 / EPSG3857 类型的简单运算可提速 2–3×。
